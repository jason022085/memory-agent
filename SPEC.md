# **半導體生產線代理之雙軌記憶更新機制與區域共享架構研究報告**

1. ## **企業級生產環境下的大型語言模型代理與記憶演進**

在人工智慧技術迅速普及的企業級生產環境中，大型語言模型（Large Language Model, LLM）代理正經歷從無狀態（Stateless）互動向具備長期狀態（Stateful）持續學習能力的典範轉移。傳統的無狀態代理在每次對話重置時，皆會遺忘先前的互動脈絡，這迫使使用者必須反覆提供相同的背景資訊，不僅嚴重影響使用者體驗，更限制了代理在複雜、多步驟的長期任務中的應用潛力 1。

另一方面，標準的RAG架構雖然能夠以外部向量資料庫引入領域知識，但對於代理與使用者之間動態生成的「互動歷史」與「工具呼叫（Tool Call）紀錄」，卻缺乏系統性的記憶更新與生命週期管理機制。在企業級高併發的生產線上，代理必須能夠清楚地區分不同課別（Section）的數據、保留所有的聊天對話（Session）軌跡。為了提升 Fab 內的協作效率，不同「課」（Section，代表公司的一個運作單位）的記憶紀錄必須達到物理級別的絕對隔離，但同一課內的工程師應能共享彼此的故障排除經驗，以確保知識的快速流動。

為徹底解決即時回應速度與長期記憶品質之間的矛盾，2026年的前沿人工智慧記憶研究提出了解耦（Decoupling）的設計理念。本報告將基於LangChain與LangGraph的狀態編排框架，搭配 Google Gemini 2.5 Flash 的推理引擎，並深度融合2026年最新的LightMem與REMem架構理論，為企業級代理設計一套「雙軌記憶更新機制（Dual-Track Memory Mechanism）」。該機制包含極低延遲的「即時軟更新（Real-time Soft Update）」與負責高階認知整合的「睡眠更新（Sleep Update）」，並採用日誌型記憶（Log-based Memory）作為底層資料結構，以確保所有對話與工具操作的絕對可追溯性。

2. ## **雙軌記憶架構之認知科學基礎與前沿理論**

當前LLM代理的記憶系統設計已不再侷限於單純的資料庫存取，而是深度借鑒了認知心理學與神經科學的理論模型。2026年的多項頂級會議論文指出，將記憶的提取、寫入與長期整合進行模組化解耦，是突破代理運算瓶頸的核心關鍵。

### **Atkinson-Shiffrin記憶模型與LightMem架構**

LightMem架構的設計深受認知科學中著名的阿特金森-希弗林（Atkinson-Shiffrin）人類記憶模型啟發。該模型將人類記憶分為感官記憶（Sensory Memory）、短期記憶（Short-Term Memory, STM）與長期記憶（Long-Term Memory, LTM）三個階段 6。在具體實踐中，這三個階段被巧妙地映射到代理的運算流程中。首先，代理透過感官記憶模組進行前置壓縮（Pre-compression），過濾冗餘 Tokens。接著，這些資訊會進入短期記憶結構中進行初步整合。

LightMem 最核心的貢獻在於徹底分離了「高頻線上操作」與「離線長期整合」。研究指出，檢索型記憶雖然輕量但充滿雜訊，而依賴大型模型驅動的線上操作則昂貴且耗時。因此，LightMem 採用高效的 Flash 等級模型來接管查詢控制與向量篩選，從而將線上延遲降至最低。而運算密集的長期記憶鞏固，則留待「睡眠時間（Sleep Time）」進行離線的批次更新，使系統能持續演化。

### **情節記憶推理與REMem架構**

REMem（Reasoning with Episodic Memory in Language Agent）架構將代理的情節記憶（Episodic Memory）推理能力推向了新高度。現有代理往往偏重語意記憶（事實與概念），卻忽略了具備時空脈絡與事件順序的情節記憶。REMem 提出結合離線索引（Offline Indexing）與線上推理（Online Reasoning）的雙階段框架。

在離線階段，REMem 會將原始對話轉換為具備時間感知（Time-aware）的結構，提取事件要旨並組織成記憶圖譜（Memory Graph）。在線上推理階段，代理配備了代理式檢索器，在圖譜上進行迭代式檢索與多步推理。在半導體場景中，這意味著 Agent 能記住特定機台在過去一段時間內的故障演變過程，而不僅僅是當下的數據。

### **軟更新與硬更新之運算權衡**

表一詳細比較了軟更新與硬更新在多個維度上的差異。

| 比較維度 | 即時軟更新 (Soft Update) | 睡眠/硬更新 (Sleep / Hard Update) |
| :---- | :---- | :---- |
| **觸發時機** | 每一輪對話或工具呼叫後即時觸發 | 系統閒置時、排程時間點或對話結束後 |
| **核心操作** | 僅附加（Append-only）原始日誌與工具執行結果 | 跨使用者提取、去重、語意壓縮、衝突排解與結構化覆寫 |
| **延遲影響** | 極低（毫秒級），非阻塞背景 I/O 操作 | 高（數秒至數分鐘），運算密集型任務 |
| **處理衝突方式** | 容許不同成員的事實矛盾並存，保留完整脈絡 | 基於時間戳記與全課語意進行邏輯仲裁，固化為 SOP |
| **API 成本** | 極低，僅需單次快速的向量嵌入（Embedding）計算 | 較高，需依賴高階 LLM 進行大範圍跨日誌分析與生成 |
| **記憶層級** | 短期記憶（STM）、原始情節日誌（Raw Logs） | 長期記憶（LTM）、區域型共享知識 (Shared Knowledge) |

3. ## **區域共享資料隔離與向量資料庫綱要設計**

在企業級生產線上，本系統採用「區域共享（Section-based Shared）」架構。

### **命名空間與硬隔離**

為了達到隔離標準，本報告建議採用「區域硬命名空間（Hard Namespace）」。具體作法是針對每個單位的課名（Section），在向量資料庫中配置獨立的集合（Collection）。命名規則為 `namespace = f"area_{section_id}"`。在這種架構下，查詢操作被嚴格限制在該課名範圍內，從物理層面徹底杜絕了不同課名記憶之間的交叉檢索，同時實現了課內不同工程師之間的經驗共享。

### **日誌型記憶之綱要結構建模**

表二展示了該日誌型記憶在向量資料庫中標準化的元資料綱要設計：

| 欄位名稱 (Field) | 資料型別 (Type) | 說明與核心用途 (Description & Purpose) |
| :---- | :---- | :---- |
| id | UUID | 記憶條目的全域唯一識別碼。 |
| vector | Float Array | 內容的語意向量嵌入，用於核心的語意相似度檢索。 |
| content | Text | 原始對話內容、工具呼叫參數、執行結果或固化後的摘要。 |
| user\_id | String | 記錄發言的工程師帳號，便於追溯與共享背景。 |
| section\_id | String | 課名單位，確保檢索範圍精確收斂，防止跨單位的知識混淆。 |
| session\_id | String | 當前對話會話的識別碼，用於串聯對話軌跡。 |
| timestamp | Integer | Epoch時間戳記，用於高效的時序查詢。 |
| role | String | 發言者角色 (user, assistant, tool, system)。 |
| memory\_type | String | 標記為 raw\_log (即時) 或 consolidated (睡眠後固化)。 |

## **4\. 系統架構與時序模型**

系統分為三層：設備數據層、即時推論平面（Hot Path）與非同步背景平面（Sleep Path）。

### **Hot Path (即時路徑)**
1. 工程師 A 在黃光 01 線 (PHOTO_LITHO_01) 詢問設備異常。
2. Agent 檢索該區域的所有歷史記憶（包含他人留下的日誌與固化知識）。
3. Agent 提供診斷建議，並透過 `soft_update_node` 即時寫入 `raw_log`。

### **Sleep Path (睡眠路徑)**
1. 背景工作者定期觸發，撈取該區域 (Section) 的所有 `raw_logs`。
2. 使用 Gemini 2.5 Flash 進行跨使用者、多日誌的深度反思。
3. 將碎片的經驗固化為 `consolidated` 記憶，並清理原始日誌。

## **5\. 結論**

本區域共享記憶架構完美對應了半導體廠「課」級別的運作實務。透過將隔離層級鎖定在 Section，Agent 不僅能保護各課的資料主權，更能作為橋樑，將課內資深與資淺工程師的經驗無縫連結。搭配 Gemini Flash 的極速推論，為 Fab 生產線提供了一套具備「團隊大腦」特質的 AI 助手。
